Hugo: Making Simple Static Site

Aug 21, 2019
8 min read
blogGitHubHugo

Introduction

In previous post I mentioned that this blog is powered by GitHub Pages and Hugo static site generator. So let’s see how easy it is to make this running and some handy info about how make it on Free GitHub services.
There are many popular static site generators out there like Next.js, Jekyll, Gatsby, Hexo to name a few but upon closer examination I decided to go with Hugo becuse I like hassle free aspect of it!

Hugo

I decided to with Hugo Extended x64 version which I downloaded from GitHub releases page. Save downloaded zip file somewhere on your system and extract it.
Now it is time to create our folder structure where we will be building our future site and Hugo framework can operate. First decide where you will be keeping your files locally. I went for D:\HUGO as base of my folder structure but off course you can place it anywhere it suits you.
Insde create two folers bin and Sites. Your directory structure should look like this:

Directory of D:\HUGO

08/20/2019  08:59 PM    <DIR>          .
08/20/2019  08:59 PM    <DIR>          ..
08/20/2019  03:25 PM    <DIR>          bin
08/20/2019  08:59 PM    <DIR>          Sites

bin folder will be used to house our downloaded executable and Sites folder is where our data for websites will live and this function as expected folder structure for hassle free operation. Now move all files from downloaded archive to bin folder.
You might want to rename EXE file from archive to somethign more friendly like hugo.exe to save on typing in future!
Your bin directory shoul now look like this:

Directory of D:\HUGO\bin

08/20/2019  03:25 PM    <DIR>          .
08/20/2019  03:25 PM    <DIR>          ..
08/17/2019  07:56 PM        92,424,710 hugo.exe
08/17/2019  07:42 PM            11,357 LICENSE
08/17/2019  07:42 PM            11,235 README.md

Now it is good time to add Hugo executable to out PATH so we can launch commands easily. If you in some time in future decide to change location of hugo.exe on your system, you might want to repeat this step or edit your PATH variable.
Now open terminal and navigate to bin folder by typing:

cd D:\HUGO\bin

We are going to add hugo.exe to our PATH environment variable so system know where to search for it when asked. Copy-paste this code and hit Enter.

set PATH=%PATH%;D:\HUGO\bin

Mow you should be all set to use Hugo. You can run it in console and explore various parameters you can set up and run it with. You can verify that it works correctly by running:

hugo version

For their detialed exlanation head over to their official documentaion. Let’s proceed to next step.

GitHub Pages

GitHub offers very nice service of hosting personal or project sites direclty from repository for free. Isn’t that neat? You are free to create your page via their provided generators or build yours from scratch. The problem is, this service does not offer you to have both, code and site content stored in same repository if you are only “free” tier user… Shame.
But on the other hand you can have private repos on free tier as well, right? Right!
So we will take advantage of this and setup our workflow accordingly. Side note, your GitHub Pages repository need to be public on free tier otherwise you will get 404 and it won’t be published by GitHub.

We will start by creating two repositories to store our content. I’m assuming you will be building personal page from now on (check GitHub Pages for instruction how to setup your organinzation o project page properly). Head over to GitHub and create repository for your page.

GitHub expects it to adhere to certain naming convention to be publicly available and properly published. It need to match this naming convention:

yourGitHubUserName.github.io

That way it will know you want to use it as your page. In my case repository name is cupsster.github.io Don’t forget to mark it as public. That will be as well your public URL to acces and display your content (assuming you do not want to configure custom domain name for it). If you try that url now, you will most likely get only 404 as there is nothing commited there right now. More on that later on.

Now create another repository. This time you can set it to private (so our source data is not visible to whole world) and name it anyhow you want. For sake of simplicity you can go with something simple like blog or so.

You should end up with 2 fresh repos, public yourGitHubUserName.github.io and private blog.

Project Setup

It is time to configure our local folder structure to use our repositories. If you are starting from scratch you need to create new dummy Hugo site now. If you already have one saved somewhere else you can skip this step.
Navigate your terminal in Sites folder created earlier and run:

hugo new site dummySite

This will create empty site skeleton in Sites\dummySite, we will need this data for later.
Still in same Sites folder clone your private repository by running this command.

git clone https://github.com/yourGitHubUserName/blog.git

Now enter your cloned repository

cd yourGitHubUserName\

Let’s assume you already have made new Hugo site somewhere else on file system or you generated dummySite as above. If you generated dummy site your Sites folders should now look something like this:

Directory of D:\HUGO\Sites

08/20/2019  08:59 PM    <DIR>          .
08/20/2019  08:59 PM    <DIR>          ..
08/20/2019  07:14 PM    <DIR>          blog
08/20/2019  07:13 PM    <DIR>          dummySite

Head over to dummySite folder and copy it’s contents to our clonned blog folder. To get up to speed we are going to add some theme to start with. There is lot of nice themes already available so pick one that fit’s you the best and we are going to integrate it. I’ll be using default xxx as example. But be sure to check theme information about it’s specific options to configure to work properly.
Enter blog folder and add Anake theme as git submodule by running this commands in terminal:

cd blog\
git submodule add https://github.com/budparr/gohugo-theme-ananke.git themes/ananke

Now you need to edit config.toml and theme support to it by adding this line:

theme = "ananke"

This will tell Hugo to use that theme tmplate for your static pages. It is good time to verify that all is working properly by running this command:

hugo server

or if you went with theme support

hugo server -t "ananke"

-t flag tells Hugo to use specific theme. You can now open your browser and go to http://localhost:1313/ which is default setting for Hugo server and verify you see webpage.
If you got it working you can kill server command by pressing Ctrl+C on keyboard. Before we move on it is good time to remove gnerated public as we will replace it in next step with custom solution. Run this in temrinal to delete it, we don’t need it for next step:

rm -rf public

With generated public folder that contain our static site gone we are going to replace it by another git submodule! Guess which one :)
Run this command in terminal:

git submodule add -b master git@github.com:yourGitHubUserName/yourGitHubUserName.github.io.git public

What we just did was to replace generated static folder with our GitHub Pages submodule! This enables us to split public and private parts of our website/blog to separate repositories and still mainting full git workflow and verioning all while running “Free” tier on GitHub. Next time you regenerate your site, public foder will get fresh site data that you can commit to your public yourGitHubUserName.github.io repository and rest of your code and content from which it is generated will stay nicely private in your blog repository. Neato! Exactly what we wanted.

You are all set! For ease of manipulation with this structure you can make handy shell script that will push contents of public folder into your yourGitHubUserName.github.io repository whan you run it with preconfigured commit message when you run it.

Content

Upon configuring yout site and theme setttings in config.toml file to your heart’s liking, next thing you want to do is to actually add some new content… Still in your blog folder run:

hugo new posts/hello-world.md

This will create new file in content/posts/hello-world.md that you can edit in your favorite text editor using modified Hugo markdown syntax. Notice there is draft: true on top, which is metadata for Hugo to treat this file as draft until you change this value to true which means you are ready to publish this post.
To preview your site with drafts locally simply run:

hugo server -D

or you can launch it in filesystem watch mode

hugo server -D -w

this way you can make live changes to your file and Hugo will automatically rebuild site and refresh your browser to reflect your changes.

Ohh did I mention it is lighting fast?! Depening on project size it is miliseconds to rebuild site. For fresh projects with not so much content and advanced features it is like 7-12ms for refresh!

Don’t forget to comit your changes and content to see your page updated.

Done

Now you can go visit your new https://yourGitHubUserName.github.io site and it only took couple of minutes to setup and it is conpletely free while you still having full control over code and content.
